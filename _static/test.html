Record Types

Inductive types can also represent record types.

Inductive plane : Set := point : Z->Z->plane.

When a constructor of an inductive type is a function, the left-hand side of a pattern patching clause can be a pattern, an application of the constructor to variables. These variables can be used in the right-hand side of the clause. In this sense, the clause is a binding construct.

Definition abscissa (p:plane) : Z :=
  match p with point x y => x end.

By using \texttt{Record}'', it is possible to define inductive types in such a way that the accessor functions are defined at the same time while the interpretation of the various fields of the record is more explicit. The following defines an inductive type \texttt{plane} with a constructor \texttt{point} and two functions \texttt{abscissa} and \texttt{ordinate} of type \texttt{plane->Z}.

Reset plane.

Record plane : Set := point {abscissa : Z; ordinate : Z}.

Records with Variants

An inductive definition can define data structures where data can have several forms and each form using different fields.

Inductive vehicle : Set :=
| bicycle : nat->vehicle
| motorized : nat->nat->vehicle.

Check vehicle_ind.

The pattern matching construct enables us to construct functions that perform different computations depending on the constructor that produced the data being observed.

Definition nb_wheels (v:vehicle) : nat :=
  match v with
  | bicycle x => 2
  | motorized x n => n
  end.

Definition nb_seats (v:vehicle) : nat :=
  match v with
  | bicycle x => x
  | motorized x _ => x
  end.

Inductive Types and Equality

Two terms obtained with two different constructors are necessarily different. When describing mathematical structures, we may need to describe quotient structures with respect to an <u>equivalence relation</u> but the equivalence relation cannot be represented by the plain equality of Coq since this would lead to inconsistencies. Several theoretical solutions have been studied to make quotient structures available. For instance, <u>setoid structures</u> have been introduced to describe types equipped with an equivalence relation. However, working with these structures quickly raises efficiency problems.

Inductive htree (A:Set) : nat -> Set :=
  hleaf : A -> htree A 0
| hnode : forall n:nat, A -> htree A n -> htree A n -> htree A (S n).


Fixpoint htree_to_btree (n:nat) (t:htree Z n) {struct t} : Z_btree :=
  match t with
    hleaf _ x => Z_bnode x Z_leaf Z_leaf
  | hnode _ p v t1 t2 => Z_bnode v (htree_to_btree p t1) (htree_to_btree p t2)
  end.

Fixpoint invert (A:Set) (n:nat) (t:htree A n) {struct t} : htree A n :=
  match t in htree _ x return htree A x with
    hleaf _ v => hleaf A v
  | hnode _ p v t1 t2 => hnode A p v (invert A p t2) (invert A p t1)
  end.
